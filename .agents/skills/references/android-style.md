# Android Classroom Style

Use `ref-android` as the instructor style source. Prefer matching these patterns even when modern Android guidance would use newer APIs.

## Strong Examples

- Login and `SharedPreferences`: `ref-android/LoginShareReference/app/src/main/java/com/ptithcm/loginsharereference`.
- Simple form + `ListView` + SQLite DAO: `ref-android/DemoQLSV/app/src/main/java/com/ptithcm/demoqlsv`.
- `BaseAdapter` list rows: `ref-android/DemoQLSV/app/src/main/java/com/ptithcm/demoqlsv/adapter`.
- `SQLiteOpenHelper`: `ref-android/DemoQLSV/app/src/main/java/com/ptithcm/demoqlsv/sqlite/Dbhelper.java`.
- `GridView` + `BaseAdapter`: `ref-android/tempProject/app/src/main/java/com/ptithcm/tempproject`.
- ViewPager + bottom navigation: `ref-android/ViewPagerNNavigation` and `ref-android/tempProject`.
- WebView demo only as a reference for WebView mechanics, not as an admin parity shortcut: `ref-android/DemoWebView`.

## Java Style

Prefer:

- Fields at the top of the class for views, adapters, lists, and DAO/API helpers.
- `onCreate` with `setContentView`, `findViewById`, adapter setup, and listeners.
- Private helper methods such as `fillProductToListView`, `loadUsers`, `clearForm`, `showMessage`.
- Anonymous listener classes or simple `implements View.OnClickListener`.
- Plain model classes with fields, constructors when useful, getters and setters.
- `try/catch` around network parsing and form conversion, with `printStackTrace` and user-facing `Toast`.

Avoid:

- Deep package structures that hide simple classroom code.
- Generic repository/use-case layers unless the Android project already has them.
- Lambdas if the surrounding code uses anonymous classes.
- Overly clever JSON reflection or annotation-heavy mapping.

## XML Layout Style

Prefer:

- `ConstraintLayout` as the root when generated by Android Studio, with an inner vertical `LinearLayout` for form content.
- `TextView` labels above `EditText` fields.
- `Button` rows inside horizontal `LinearLayout`.
- `ListView`/`GridView` for repeated admin data.
- Simple drawable XML for rounded buttons and inputs.
- IDs named by widget and purpose: `edtEmail`, `btnLogin`, `lvProducts`, `spStatus`, `tvOrderNumber`.

Avoid:

- Compose UI.
- RecyclerView-first implementations unless the project already uses it and the user accepts the added complexity.
- Complex custom views for simple admin forms.

## Network Style

If no HTTP library exists in the Android app, keep network code explicit and small:

- Centralize base URL and token header in one helper class.
- Use simple request methods such as `get`, `postJson`, `putJson`, `patchJson`, `delete`.
- Parse JSON into plain models manually enough for the screen.
- Keep UI updates on the main thread.

If a library already exists, follow the local project pattern. Do not introduce Retrofit or other HTTP frameworks without user approval.

## Advanced Feature Policy

Ask before using these:

- Image picker abstractions beyond Android's basic intent flow.
- Chart libraries for dashboard revenue.
- WebSocket/STOMP support for live support chat.
- Background services, WorkManager, dependency injection, Room, or Navigation Component.

When not approved, build a simpler parity version: manual refresh, list/table display, basic image URL metadata, or explicit activity navigation.
